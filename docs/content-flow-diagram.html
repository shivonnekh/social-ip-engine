👨‍⚕️ Social IP · Content Flow Diagram

A simplified map for the production team — from an idea in Notion to a real Instagram follower getting an AI DM reply in seconds. Goal: see at a glance what a human does, what runs automatically, and where the real bottlenecks are.

🧑‍🎨 CC (Content Creator) owns this 🤖 Automated (system / AI) 🖐️ Manual review checkpoint

① The big picture: two stages, two systems

Stage 1 is "making the content" (Notion + the Studio tooling — mostly automated, with human review gates). Stage 2 is "the content goes live and auto-engages" (the real Instagram bot, social-ip-engine — fully automated). These are two separate systems, connected by one real webhook bridge.

The bottleneck isn't automation — it's judgment. The remaining manual steps are human review gates (is the image OK? is the final video OK? does the funnel reply correctly?). Everything else — keyword, DM copy, public reply, and the infographic — is fully automatic.

② Who owns what

🧑‍🎨 CC (Content Creator)

  • Comes up with the idea: Topic / Hook / CTA (the keyword to comment)
  • Writes the Shot Guide — the single source of truth for everything downstream. The more cinematic and specific it is, the better the AI-generated image and video turn out
  • Reviews the AI-generated image and final video, sends it back for a prompt tweak if it's off
  • Flips the Notion status to "🟢 Ready to Publish" — this arms the whole comment → DM funnel (keyword, DM, public reply, infographic) so it can be tested before going live
  • Publishes the final video to Instagram + flips to "✅ Published" — at this point the post just goes out, already armed

🤖 Automated (system / AI)

  • Fans one idea out into a production row per IP (Jackie / Jessica)
  • Derives the image prompt, voice script, and video prompt from the Shot Guide
  • Calls the AI image model (gpt-image-2), AI voice (MiniMax), AI video (Dreamina/即梦)
  • The moment Notion flips to "🟢 Ready to Publish", a real webhook fires and auto-drafts the keyword, DM copy, and public reply
  • Auto-attaches the infographic image — reuses one already uploaded in Notion, or generates one on the spot from the Infographic Brief if none exists
  • Persists everything (rule + image) to git, so nothing is lost on the next deploy; already-processed rows are never re-run or re-generated
  • When a real follower comments the keyword, sends the DM automatically and hands the conversation to the AI persona agent

🖐️ Manual review checkpoints (still need human eyes)

  • Is the AI-generated still frame clear and on-message enough, no weird distortion?
  • Does the final video flow well — lip-sync and expression look natural?
  • These are the only two places a human needs to look at every single piece of content

③ How the technical flow works

(a) How the image gets generated

  1. CC writes the "🎥 Shot Guide" in Notion — rich and cinematic (action, camera moves, cuts). This is the single source of truth CC
  2. The system automatically extracts just the first beat of the Shot Guide into a one-line "image prompt" (a single still frame) automated
  3. The system pulls this IP's reference face photos + a clinic background from Notion and sends them to the AI image model (gpt-image-2) automated
  4. The generated image is placed back into the right Notion block, and the row gets ticked "🎨 image done" automated
  5. A human glances at the image — is it clear enough, any distortion? manual review

(b) How it connects to the keyword rule (comment → DM)

  1. When the content is ready, CC flips the Production Tracker row's Stage to "🟢 Ready to Publish" in Notion — this arms the funnel before the post goes live CC
  2. That status change fires a real Notion Automation webhook straight to social-ip-engine's POST /admin/notion-sync — Notion pushes to us, we don't poll for it automated
  3. The system drafts the keyword, the DM copy, and the public reply, and writes it straight into the live rule file automated
  4. The rule is live immediately — no redeploy needed, Instagram can match it right away automated
  5. The infographic image is attached automatically too: if one is already uploaded into the row's "📊 DM Infographic" toggle, it's downloaded and wired in; if not, but an "Infographic Brief" exists on the content page, the system generates one on the spot (capped at 5 generations per sync run, to bound image-API spend on a bulk publish) automated
  6. CC can now test-comment their own post to confirm the DM + infographic fire correctly — before publishing. Then flip to "✅ Published" and post CC

Where it's easy to get confused: Two of the three connections are real, instant webhooks (Notion Publish → keyword sync, and Instagram → the bot). Only the very first stage — the Studio content-authoring tooling checking Notion for new work — is a polling loop, and that's the one place a few minutes of lag is expected and normal.

④ How it shows up on Instagram

1️⃣ Reel goes live

A 9:16 short video with a clear CTA — "comment '[keyword]' for more." This is the entry point for everything downstream.

2️⃣ A follower comments the keyword

The system gets Meta's webhook notification instantly, matches the comment text against the keyword rule table — a match sends an automatic DM right away, and optionally a public acknowledgement reply on the comment thread.

3️⃣ No keyword match = silence

This is intentional — the system will never DM a stranger just because they left any comment. No match, no reply, by design.

4️⃣ The conversation continues in DM

Once the follower replies in DM, it's no longer a canned message — the AI persona agent (Chloe or Jackie) takes over, referencing prior chat history, and keeps the conversation going like a real person would.

⑤ What shipped on 2026-07-03 — the pre-flight upgrade

The auto-sync existed but had never actually run in production (three config values were missing on the server). It's now live, hardened, and moved to a pre-flight model. Here's what changed.

(a) One flip to 🟢 Ready to Publish wires the whole funnel

A single webhook writes one complete comment → DM rule. Nothing here is manual.

FieldWhat it doesWhere it comes from (auto)
keywordThe word a follower comments to trigger the replyThe row's CTA
dm_textThe private DM sent, in the IP's voice (Jackie / Chloe)The row's First-DM copy
image_urlsThe infographic attached to the DMThe "📊 DM Infographic" toggle, or auto-generated from the Brief
public_ackThe public reply posted on the comment thread ("sent you a DM 👀")Auto-templated
accountsWhich IP account the rule applies toThe IP relation

(b) Infographics generate themselves — no manual upload needed

🖼️ Image already in Notion

If you dropped a PNG into the row's "📊 DM Infographic" toggle, it's downloaded and wired in as-is.

🤖 No image? Generated from the Brief

If the row has an "Infographic Brief" but no image, the system generates one on the spot (gpt-image-2) from that Brief and attaches it. Capped at 5 generations per sync run to bound spend on a bulk publish.

🖐️ Why pre-flight matters here

Because arming happens at Ready to Publish, you get to look at the auto-generated image and test the DM before the post is live — instead of discovering a bad image after it's already replying to strangers.

(c) Bounce the stage all you like — it never re-runs

Drag a row Ready → Idea → Ready again? It will not re-wire or re-generate. Three independent guards make "already done stays done" safe:

  1. Row-level — every processed row is recorded in a state file that's saved to git, so it survives redeploys. A re-flip is skipped on the very first check main guard
  2. Keyword-level — even if that state were lost, a rule for the same keyword already existing is skipped, so no duplicate backup guard
  3. Image-level — even if both above failed, an already-generated PNG is reused, never re-generated, so it never re-spends on the image API cost guard
The only time a row re-wires is if you deliberately delete its rule or change its keyword — i.e. when you actually want a fresh one, never by accident.

(d) Durability — nothing vanishes on deploy anymore

The server runs on ephemeral disk: a redeploy rebuilds from GitHub. Previously an auto-wired rule (and its image) could disappear on the next unrelated deploy. Fixed:

  • Auto-wired rules + the infographic PNGs are now committed and pushed to git by the server itself, so they persist across every deploy.
  • Three server config values that were silently missing — the webhook auth secret, the Notion API key, and the git push token — are now set, so the whole loop actually runs and self-persists.
  • A stale developer tunnel URL that was breaking every DM's image link was corrected to the real production URL.

💡 Two things worth discussing with the team

  1. Image / video review is still 100% eyeballed one by one. Worth defining a simple "obviously reject" checklist (face distortion, message clarity) to cut down back-and-forth revision rounds manual checkpoint
  2. Notion → studio tooling is still polling (a few minutes of lag before a new concept gets picked up for fan-out). If the team wants that tighter, it's worth discussing switching it to a real trigger too, the same way the Publish step already works
