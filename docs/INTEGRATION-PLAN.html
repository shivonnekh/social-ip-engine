2. Diagnosis — why it feels fragmented

The two-layer reply system you want already exists and is live — canned keyword replies for comments, a persona agent for DMs. What's broken is configuration sprawl and the disconnect from the content factory. Verified in code:

Concrete cost Onboarding one new IP today = edits in 5+ hardcoded dicts across two repos, new env var names, a Notion row, and a persona JSON — with nothing enforcing consistency. Adding a platform (Messenger, TikTok) multiplies the same sprawl.

2.3 The content → reply bridge is half-manual

  • automated POST /admin/notion-sync drafts a keyword rule when a Production Tracker row flips to ✅ Published.
  • manual Infographics generated in ai-tcm-ip (server/static/infographics/<brand>/<keyword>.png) must be hand-copied into TCM-Jessica's data/media/guides/ and hand-wired into image_urls.
  • duplicated scripts/notion_ids.json is copy-pasted between repos and kept in sync by memory.

2.4 Dead weight

  • ai-tcm-ip/server/ (webhook.py, dm_map.json, its render.yaml) — never was the live system; caused a real "which repo is live?" incident on 2026-07-01. Confirmed 2026-07-03: no zombie Render service exists — but the files still mislead every future session. Archive them.
  • campaigns/ in ai-tcm-ip mixes two naming conventions (numbered 01-tonsil-stone and slug-based).
  • The legacy WhatsApp stack (deep pipeline, sales/booking tools, proactive broadcaster) belongs to a discontinued line of business but is still deployed — see Decision D2.

3. Target architecture — two reply layers, many platforms

Design principle: one IP, one brain, many platforms. Reply depth is fixed at two layers; platforms plug in as thin adapters around the same shared core (meta_webhook.py already works this way for IG + FB).

LayerSurfaceEngineCost/turnStatus
L0 · Canned Comments comment_rules.py — keyword match → private reply + infographic + public ack. No LLM. Since 2026-07-02: bare-keyword-only in DMs + per-user dedup (guides_sent) — genuine questions always reach L1. $0 live
L1 · Persona DMs ChloeAgent → generalize to PersonaAgent. One LLM call: per-IP persona prompt + last 16 CRM messages. No tools, no sales machinery, no off-platform push. ~$0.001 live; needs config-driven multi-IP
What happened to the deep pipeline? The Planner→Specialists→Writer stack, sales tools, booking flow, and broadcaster stay in the codebase but are parked: not part of the reply strategy, not extended, not refactored. The dormant SOCIAL_PIPELINE_ACCOUNTS escape hatch (KB-grounded DM replies with automatic fallback to PersonaAgent) also stays as-is — zero cost, tested fallback net. If a future strategy needs depth again, it's there; until then it is code, not product.

5. The IP Registry — single source of persona truth

One folder per IP under data/ips/. Every hardcoded dict from §2.1 is replaced by a loader that reads these files. Notion's 👤 IP Registry stays the human authoring UI; a sync script materializes it into versioned JSON (Notion → git, one direction).

data/ips/
├── jackie/
│   ├── ip.json          # identity + wiring (below)
│   └── persona.json     # DM system prompt, greetings (today's data/personas/jackie.json)
└── chloe/

ip.json — the complete wiring for one IP

{
  "id": "jackie",
  "display_name": "Jackie Chan TCM",
  "language": "en",                        // drives comment-rule language gate
  "active": true,

  "channels": {                            // ← the platform expansion axis
    "instagram": {
      "account_id": "17841417304649448",
      "token_env": "IG_TOKEN_JACKIE",      // replaces meta_client.py hardcoded map
      "comments": "canned",                // L0 on
      "dms": "persona"                     // L1 on ("off" | "persona" | "pipeline")
    },
    "messenger": { "page_id": "…", "token_env": "FB_TOKEN_JACKIE",
                   "comments": "canned", "dms": "persona" },
    "tiktok":    { "enabled": false }      // future — same shape
  },

  "content": {                             // consumed by studio/ scripts
    "voice_id": "elderly_man",
    "speed": 1.2, "pitch": 0,
    "tts_language": "English",
    "notion_ip_page": "<IP Registry row id>",
    "infographic_brand_slug": "jackie"
  }
}
Payoff Adding an IP or lighting up a platform for an existing IP becomes: 1 Notion row / 1 config block + tokens in Render env. Zero code changes. The loader validates every ip.json at startup — a malformed IP can't take down the others.

6.3 Render config — keep content commits from redeploying the bot

# render.yaml addition
services:
  - type: web
    name: social-ip-engine
    ...
    buildFilter:
      paths: [src/**, data/**, render.yaml, pyproject.toml]
      ignoredPaths: [studio/**, docs/**]
Landmine — learned the hard way When touching render.yaml, IG_ENABLED / FB_ENABLED MUST stay sync: false — a literal value: gets re-applied by blueprint sync on every deploy and silently kills all IG processing (the 2026-07-01/02 multi-day outage). Auto-deploy webhook is broken anyway — deploys go through the Render API (key now in .env as RENDER_API_KEY).

7. Content → reply bridge — full automation

Today (half-manual)

  1. Notion row → ✅ Published
  2. notion-sync auto-drafts keyword rule ✅
  3. Infographic: hand-copy PNG between repos ✋
  4. Hand-edit image_urls in the rule ✋
  5. Spot-check rule language ✋ (mechanical templating gotcha)
  6. dm_map.json drifts as a third copy ✋

Target (one-touch)

  1. Notion row → ✅ Published
  2. notion-sync drafts the rule ✅
  3. …AND pulls the infographic from the row's 📊 DM Infographic toggle → saves to data/media/guides/ → wires image_urls automatically ✅ (same repo now — a file copy, not a cross-repo ritual)
  4. …AND runs a language-consistency check → flags mismatches to the sync report instead of shipping them ✅
  5. dm_map.json + export_dm_map.py deleted — Notion is the only authoring source ✅

Everything extends the existing src/notion_sync.py (stdlib-only, idempotent). The only reason step 3 was impossible before is that the PNG lived in a different repo.

8. Migration phases

Ordered so production is never bet on a refactor. Every phase is independently shippable and leaves the full suite green (758 passing as of 2026-07-03).

Phase 0 — Safety & cleanup

~half a day · zero risk
  • Tag both repos: pre-integration-2026-07. Snapshot production Postgres (note: free-tier 90-day expiry — verify date).
  • Baseline probe: signed synthetic webhook → confirm status is not disabled.
  • Resolve Decision D2 (legacy WhatsApp services on/off) — it changes what Phase 2 archives.

Phase 1 — IP Registry inside social-ip-engine

~1 day · behavior-neutral refactor
  • Create data/ips/{jackie,chloe}/ip.json + move persona JSONs alongside; src/ips/registry.py loader with startup schema validation.
  • Replace the 4 hardcoded dicts with registry lookups. Assert-identical tests first: snapshot current dict outputs, prove the registry returns the same mappings.
  • Rename ChloeAgent → PersonaAgent (constructor takes an IP record; behavior byte-identical).
  • persona_dry_run_social.py + full pytest before deploy.

Phase 2 — Repo merge + dead-code archive

~1 day · no production code changes
  • Git subtree merge per §6.4 → studio/.
  • Archive to docs/legacy/: server/webhook.py, dm_map.json, export_dm_map.py, ai-tcm-ip's render.yaml. Delete server/.
  • Normalize campaign folder naming (slug convention — D4).
  • Single studio/notion_ids.json; point src/notion_sync.py at it.
  • Add buildFilter (§6.3 — mind the IG_ENABLED landmine). Manual deploy + probe after.
  • Merge the two CLAUDE.md files; archive the ai-tcm-ip GitHub repo with a pointer README.
  • Optional: move parked WhatsApp stack → src/legacy/ (only if mechanical + green).

Phase 3 — Bridge automation

~1 day
  • Extend notion_sync.py: pull 📊 DM Infographic → data/media/guides/<keyword>-page-1.png → auto-wire image_urls.
  • Language-consistency check on drafted rules (flag, don't block).
  • sync_ips.py: Notion IP Registry → data/ips/*/ip.json, one-way, git-reviewed. Studio scripts read voice/face config through it.
  • Retire the manual copy step from the runbook.

Phase 4 — Messenger activation

~1–2 days (most code exists)
  • facebook.py adapter + shared meta_webhook core already handle FB events — wire Jackie's FB Page into ip.json → channels.messenger, tokens in Render env.
  • Comment rules: reuse the same keyword rules (schema already platform-agnostic via platform field on events).
  • CRM key namespacing: verify fb_ vs ig_ prefixes keep users distinct per platform.
  • End-to-end: FB comment → canned DM; FB DM question → PersonaAgent reply; synthetic probe for FB_ENABLED.

Phase 5 — TikTok (scoping first)

scope: ~half a day research before committing
  • TikTok's official APIs for comment management + DMs are far more restricted than Meta's — scope what's actually possible (Business API comment replies exist; DM automation is limited/regional).
  • Design the adapter to the same shape as instagram.py/facebook.py; content side already produces 9:16 video natively.
  • Decide: official API vs. manual-assisted workflow (agent drafts replies, human posts) if API access is blocked.

10. Risks & what NOT to do

RiskSeverityMitigation
IG/FB processing silently dies via IG_ENABLED/FB_ENABLED blueprint reset on render.yaml changesHIGHKeep sync: false; synthetic-webhook probe after every deploy (now scripted); on the phase checklists.
Repo merge confuses the Render buildMEDstudio/ has its own requirements.txt, never referenced by the service build; buildFilter excludes it; Phase 2 ends with manual deploy + probe.
Canned keyword rules hijack genuine DM questions (2026-07-02 incident)—fixed + deployed: bare-keyword-only trigger (EN + CJK) + guides_sent dedup + 29 regression tests. Residual: guides_sent read-modify-write race under duplicate webhook delivery — cleanup in Phase 1.
Backfill scripts re-sending old guides (04:44 mass re-send, trigger still unidentified)MEDInvestigate before running any backfill again; dedup now limits blast radius to one send per user per keyword.
Notion IP Registry and ip.json driftLOWOne-way sync + git diff review; ip.json is what production reads, so drift is visible.
Postgres free tier expiryMEDCheck expiry in Phase 0.

Do NOT

  • Do not extend or build on the parked WhatsApp stack — it's code, not product.
  • Do not let any persona mention clinics, bookings, prices, other personas, or push users off-platform — policy lives in the persona files, not in people's memories.
  • Do not remove the PersonaAgent fallback net around the dormant pipeline path — tested safety floor.
  • Do not create a new Render service or move webhook URLs — every Meta subscription would need re-verification.
  • Do not let studio scripts import from src/ or vice versa except via the two sanctioned seams: data/ips/ (registry) and notion_sync.py (publish bridge).
  • Do not hardcode any IP's account id, token name, or language in Python — if you're typing an account id into a .py file after Phase 1, the design has failed.
  • Do not run local backfill scripts for anything conversational — POST /admin/backfill-comments (real CRM) only.
